Lezione 2 — Codificare gli algoritmi con i Flowchart

Obiettivo: rappresentare algoritmi in modo grafico, chiaro e non ambiguo usando i simboli standard dei diagrammi di flusso.

Che cos’è un Flowchart e a cosa serve

  • È una rappresentazione grafica di un algoritmo: blocchi (azioni/decisioni) + frecce (flusso di controllo).
  • Favorisce comprensione, revisione tra pari e traduzione in pseudocodice/codice.
  • Utile prima della programmazione, per progettare e comunicare la logica.

Standard e convenzioni. I simboli dei flowchart provengono da standard storici (ad es. ISO 5807, oggi ritirata, e DIN 66001) e da convenzioni ANSI/ISO diffuse. Nella pratica basta adottare un set di simboli uniforme e coerente per tutto il corso/documento.

Simboli fondamentali

  • Terminator (ovale): Start / End.
  • Process (rettangolo): istruzione/azione (x ← x+1).
  • Input/Output (parallelogramma): leggi/scrivi (leggi n, stampa s).
  • Decision (rombo): condizione con rami Sì/No.
  • Predefinito / Sottoprocesso (rett. con barre laterali): “chiama procedura”.
  • Connettori: cerchio (on-page) / “bandiera” (off-page) per evitare linee incrociate.
  • Frecce: direzione del flusso; preferire top-down, sinistra→destra.

Mappa concettuale: costrutti ↔ simboli

  • Sequenza → Process in catena.
  • Se/Altrimenti → Decision con due uscite Sì/No che si ricongiungono.
  • While → test in alto che, se Sì, entra nel corpo e poi torna al test.
  • Repeat…Until → corpo prima, test in basso; No ripete, Sì esce.
  • For → init → test → corpo → incremento → ritorno al test.
IF / ELSE
WHILE (pre-condizione)

Il ritorno al test è ora una spezzata che va a sinistra, sale, rientra sul test dall’alto: nessuna intersezione.

REPEAT … UNTIL (post-condizione)

Corpo ben distanziato dal test: No ripete, Sì termina.

FOR (init → test → corpo → incremento)

Regole di stile (leggibilità & correttezza)

  • Un solo Start e uno o pochi End; evita rientri multipli “misteriosi”.
  • Frecce chiare, niente incroci; usa connettori se il diagramma cresce.
  • Decisioni binarie con etichette Sì/No; testi brevi e precisi nei blocchi.
  • Granularità: ogni Process fa una cosa chiara; sposta dettagli in sottoprocessi se necessario.
  • Coerenza tra flowchart, pseudocodice e specifica I/O.

Esempio 1 — Massimo tra due numeri

Specifica: input a,b interi; output max.

  1. Start → I/O: leggi a,b
  2. Decision: a ≥ b ?
  3. Se Sì: Process max ← a → vai a End
  4. Se No: Process max ← b → vai a End
  5. Output: stampa max → End
leggi a, b
se (a >= b) allora
    max ← a
altrimenti
    max ← b
stampa max

Esempio 2 — Somma di una sequenza con sentinella

Specifica: leggere interi finché non arriva -1; stampare la somma.

  1. Start → Process: somma ← 0
  2. Input: leggi x
  3. Decision: x = -1 ?
  4. Sì → Output stampa somma → End
  5. No → Process somma ← somma + x → ritorna a “leggi x”
somma ← 0
ripeti
    leggi x
    se (x ≠ -1) allora somma ← somma + x
finché (x = -1)
stampa somma

Esempio 3 — Massimo di N valori (contatore)

Specifica: dati N e poi N numeri, stampare il massimo.

  1. Start → leggi N → i ← 0
  2. Decision: i = 0 ? (primo giro) → se Sì leggi x e max ← x, i ← 1
  3. Loop: finché i < N: leggi x → se x > max allora max ← x → i ← i+1
  4. Output stampa max → End
leggi N
per i da 1 a N:
    leggi x
    se (i = 1) allora max ← x
    altrimenti se (x > max) allora max ← x
stampa max

Esercizi (show/hide soluzione)

  1. Pari/Dispari: leggi un intero e stampa “PARI” se n mod 2 = 0, altrimenti “DISPARI”.
  2. Conteggio positivi: leggi N e poi N numeri; conta quanti sono > 0.
  3. Fattoriale: dato n ≥ 0, calcola n! con un ciclo.
Start → leggi n → Decision: (n mod 2 = 0?) 
  sì → stampa "PARI" → End
  no → stampa "DISPARI" → End

leggi N; c ← 0; i ← 0
mentre (i < N):
    leggi x
    se (x > 0) allora c ← c+1
    i ← i+1
stampa c

leggi n; f ← 1; i ← 2
mentre (i ≤ n):
    f ← f * i
    i ← i+1
stampa f

Errori tipici da evitare

  • Decisioni con >2 uscite senza etichette chiare (rami non deterministici).
  • Linee che si incrociano inutilmente (usa connettori).
  • Blocchi “onnicomprensivi” (mancanza di granularità) → difficile tradurre in codice.
  • Output prima di aver gestito tutti i casi (rami “orfani”).

Riferimenti essenziali

  • Simboli de-facto ANSI/ISO per diagrammi di flusso (es. ISO 5807 — ritirata; DIN 66001).
  • Qualsiasi manuale introduttivo di algoritmica/FP che adotti la notazione standard dei flowchart.
